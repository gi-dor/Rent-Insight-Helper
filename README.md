# 🏠 Rent Insight - 전월세 실거래 기반 시세 조회 및 분석 서비스 

## 1. 기획 배경
서울에서 전세 매물을 구하는 과정에서, 대부분의 부동산 플랫폼이 제공하는 `부정확한 시세 정보와 허위 매물` 로 인해 직접 부동산에 연락하며 시세를 파악해야 하는 불편함을 겪었습니다.<br>
이 경험을 통해, **`실제로 거래된 데이터를 기준으로, 내가 원하는 지역의 정확한 전월세 가격을 알 수는 없을까?`** 라는 개인적인 필요에 의해 이 프로젝트를 시작하게 되었습니다.

## 2. 프로젝트 개요
**Rent Insight**는 국토교통부의 공공데이터를 기반으로, 서울시 오피스텔 전월세 실거래 데이터를 매일 자동으로 수집하고<br>
사용자가 원하는 조건에 맞는 실거래가 정보를 조회하고 분석할 수 있는 백엔드 중심의 웹 애플리케이션입니다.

단순한 CRUD를 넘어, **외부 API 연동, 데이터 수집 자동화, 대용량 데이터 조회 성능 최적화, 그리고 Spring Security를 이용한 인증/인가 처리**에 중점을 두어 개발을 진행했습니다.

[비즈니스/서비스 측면]
- 100% 신뢰 가능한 데이터 : 허위 매물이 아닌, 검증된 공공 실거래 데이터만을 제공
- 직관적인 시세 파악 : 평형, 보증금, 월세 등 세부 조건에 따른 지역별 실제 체감 물가 확인

[기술적 측면]
- 외부 데이터 연동: 실무에서 자주 사용되는 공공데이터 수집 및 처리 구조 경험
- 자동화 설계 : Spring Scheduler 기반 정기적인 배치성 수집 작업(Daily Batch) 설계
- 효율적인 ORM & SQL : 단순 CRUD는 JPA로, 복잡한 동적 검색 조건은 MyBatis로 병행 사용

## 3. 기술 스택 (Tech Stack)

### Backend
- **Language**: `Java25 -> Java 11`
- **Framework**: `Spring Boot 4.x`, `Spring Security`
- **ORM**: `JPA (Spring Data JPA)`
- **SQL Mapper**: `MyBatis`

### Database
- **RDBMS**: `MySQL 8.0`

### Frontend
- **Template Engine**: `Thymeleaf`
- **JavaScript Library**: `jQuery`, `JavaScript (ES6)`
- **CSS Framework**: `Bootstrap 5`

### DevOps & Testing
- **Scheduler**: `Spring Scheduler`
- **Performance Test**: `nGrinder 3.5`

## 5. 조회 성능 최적화: 쿼리 실행 시간 98% 단축 (45배 향상)

프로젝트의 핵심 기능인 '실거래가 검색'은, 데이터가 누적될수록 성능 저하가 발생할 수 있는 치명적인 병목 지점이었습니다. `EXPLAIN`, `nGrinder`, `MySQL Profiling`을 활용하여 문제를 진단하고, 인덱스 설계를 통해 성능을 극적으로 개선했습니다.

### 문제 진단: Full Table Scan의 발견
- **현상**: `EXPLAIN` 분석 결과, 약 17만 건의 데이터에 대해 `type: ALL` (Full Table Scan)이 발생하고 있었습니다.
- **성능 측정 (As-Is)**: `nGrinder`를 통해 99명의 동시 접속 부하 테스트 시, **99명을 초과하면 에러가 발생**하는 명확한 한계점을 확인했습니다.

### 해결 과정: 복합 인덱스 설계 및 적용
- **전략**: `WHERE` 절의 검색 패턴(동등 비교 우선, 범위 비교 후순)을 분석하여, 가장 효율적인 **`(sgg_cd, umd_nm, area)` 복합 인덱스**를 설계하고 적용했습니다.

## 6. 겪었던 문제 및 해결 과정 (Troubleshooting)

### 1. Spring Security 커스텀 로그인 실패 처리
- **문제**: Spring Security의 기본 로그인 실패 처리는, 단순히 "자격 증명에 실패하였습니다" 와 같은 획일적인 에러 메시지만을 제공합니다. 이는 사용자에게 "ID가 틀린 것인지, 비밀번호가 틀린 것인지"에 대한 명확한 피드백을 주지 못하여 사용자 경험(UX)을 저해하는 문제점이 있었습니다.
- **해결**:
  1. `AuthenticationFailureHandler` 인터페이스를 구현하는 **`LoginFailHandler`** 클래스를 직접 생성했습니다.
  2. `onAuthenticationFailure` 메서드를 오버라이드하여, 로그인 실패 시 발생하는 `AuthenticationException`의 구체적인 타입을 확인(`instanceof`)하는 분기 로직을 구현했습니다.
  3. **`UsernameNotFoundException`** 발생 시 "존재하지 않는 아이디입니다.", **`BadCredentialsException`** 발생 시 "비밀번호가 일치하지 않습니다." 와 같이, 각 예외 상황에 맞는 구체적인 에러 메시지를 설정했습니다.
  4. 이 에러 메시지를 `RedirectAttributes`에 담아 로그인 페이지로 리다이렉트 시킴으로써, Thymeleaf 뷰에서 사용자에게 실패 원인을 명확하게 보여주도록 설계했습니다.
  5. 마지막으로 `SecurityConfig`에 `.failureHandler(loginFailHandler())`를 등록하여, Spring Security가 기본 핸들러 대신 직접 구현한 커스텀 핸들러를 사용하도록 설정했습니다.
- **결과**: 이 커스터마이징을 통해, 사용자는 로그인 실패 원인을 명확하게 인지하고 다음 행동을 결정할 수 있게 되어, 서비스의 사용 편의성을 크게 개선할 수 있었습니다.

> **자세한 구현 과정은 아래 기술 블로그에 기록했습니다.**

### 2. MyBatis 동적 쿼리 및 페이징 구현의 복잡성
- **문제**: 지역, 보증금, 평형 등 사용자의 검색 조건은 매우 다양하며, 어떤 조건이 들어올지 예측할 수 없었습니다. 이를 Java 코드의 `if-else` 분기문으로 처리하는 것은 코드의 복잡도를 기하급수적으로 증가시키는 문제점이 있었습니다.
- **해결**:
  1. **동적 쿼리**: MyBatis의 `<if>`, `<choose>`, `<when>` 등 동적 SQL 태그를 적극적으로 활용하여, DTO의 필드 값이 `null`이 아닐 경우에만 `WHERE` 절에 `AND` 조건이 동적으로 추가되는 유연한 쿼리를 구현했습니다.
  2. **파라미터 전달**: 여러 종류의 파라미터(검색조건 DTO, 페이징 변수)를 전달하기 위해, `@Param` 어노테이션을 사용하여 각 파라미터의 이름과 타입을 명확히 지정하여 코드의 가독성과 타입 안정성을 크게 향상시켰습니다.

### 3. 프론트엔드 '상태 관리' 문제 (쿠키 구현)
- **문제**: 모달 팝업에 "24시간 동안 다시 보지 않기" 기능을 구현하기 위해, 사용자의 선택을 브라우저에 저장해야 했습니다.  순수 프론트엔드 레벨에서 상태를 관리 했습니다.
- **해결**:
  1. **쿠키 활용**: 브라우저의 쿠키(Cookie)를 해결책으로 선택하여, `setCookie()` 헬퍼 함수를 통해 '유효기간이 24시간인' 쿠키를 생성하도록 구현했습니다.
  2. **엣지 케이스 고려**: 단순히 `document.cookie.includes()`를 사용하면, `do_not_hideNotice`와 `hideNotice`처럼 이름이 유사한 다른 쿠키에 의해 오작동할 수 있다는 점을 인지했습니다. 이를 방지하기 위해, 쿠키 문자열을 `;`로 분리하고, 정확한 `key`로 시작하는지 검사하는 안정적인 `getCookie()` 헬퍼 함수를 직접 구현하여 버그 발생 가능성을 원천 차단했습니다.

### 4. 개발/테스트 환경 구성 문제
- **문제**: 부하 테스트 도구인 `nGrinder` 실행 시, `ClassNotFoundException`이 발생하며 Worker Process가 시작되지 않았습니다.
- **해결**: 원인이 nGrinder 3.5.x 버전과 최신 Java(17+) 버전 간의 호환성 문제임을 파악했습니다. 가장 안정적인 해결책은 **테스트 실행 환경을 nGrinder가 공식적으로 지원하는 Java 11로 다운그레이드**하는 것이라고 판단하고, `JAVA_HOME` 환경 변수를 재설정하여 안정적인 테스트 환경을 구축하고 부하 테스트를 성공적으로 완료했습니다.
