1️⃣ [ Rent-Insight-Helper ]

공공데이터 기반 전월세 실거래 시세 조회 및 분석 서비스
GitHub: https://github.com/gi-dor/Rent-Insight-Helper

프로젝트 개요

Rent Insight는 국토교통부의 공공데이터를 기반으로, 서울시 오피스텔 전월세 실거래 데이터를 매일 자동으로 수집하고 사용자가 원하는 조건에 맞는 실거래가 정보를 조회하고 분석할 수 있는 백엔드 중심의 웹 애플리케이션입니다.

단순한 CRUD를 넘어, 외부 API 연동, 데이터 수집 자동화, 대용량 데이터 조회 성능 최적화, 그리고 Spring Security를
이용한 인증/인가 처리에 중점을 두어 개발을 진행했습니다.

사용 기술

Spring Boot / MyBatis / JPA / MySQL / Spring Security / nGrinder

[ 주요 기능 ]

  • 데이터 수집 자동화
    • Spring Scheduler를 활용하여 매일 9시 국토교통부 공공데이터 API 자동 호출
    • 수집 성공 / 실패 이력을 DataUpdateLog 테이블로 관리
    • API로 조회한 데이터를 DB와 한건씩 비교하여 새로운 데이터만 INSERT하는 방식을 고려했습니다. 하지만 매일 수백 수천 건의 데이터를 비교하는 과정에서 과도한 SELECT 쿼리가 발생하여 오히려
      성능이 저하될 수 있다고 판단해서 기존 데이터 DELETE 후 Batch INSERT 방식 적용
  • 조건 기반 대용량 조회
    • 지역, 건물명, 보증금/월세 범위, 전용면적(평형) 등 복합적인 조건에 따른 동적 동적 쿼리 구현
    • 복잡한 조건이 많은 조회이므로 MyBatis로 역할 분리
    • 복합 인덱스를 설계하여 조회 쿼리 실행속도 약 45배 개선
      • 인덱스 적용 전: 0.076037 초 (약 76.0 ms)
      • 인덱스 적용 후: 0.001675 초 (약 1.7 ms)
  • 사용자
    • Spring Security를 적용하여 세션 기반 로그인/로그아웃 기능 구현
    • 회원가입 시 BCryptPasswordEncoder를 사용하여 비밀번호 암호화 후 저장
    • UserDetailsService를 구현하여 DB 기반 사용자 인증 처리
    • 로그인 실패시 SimpleUrlAuthenticationFailureHandler 를 구현하여 실패 사유에 따른 에러 메시지 처리 로직 구현

  • 팝업 모달 및 24시간 보지 않기 기능 구현
    • 메인 페이지 진입 시 서비스 업데이트
      로드맵 안내를 위한 Bootstrap 모달 팝업 구현
    • “24시간 동안 보지 않기” 체크박스 선택 시 쿠키(hideNotice) 를 생성하여 재노출 방지

왜 DELETE 후 INSERT 방식을 선택했는가?

  • 초기에는 기존 DB 데이터와 API 수집 데이터를 한 건씩 비교하여 신규 데이터만 INSERT 하려했습니다.
  • 매일 수백~수천 건의 데이터를 비교하기 위해 대량의 SELECT 쿼리가 반복적으로 발생하게 되면
    오히려 전체 수집 성능을 저하시킬 수 있다고 판단 했습니다
  • 해당 데이터는 월 단위(DEAL_YMD)로 명확히 구분되는 특성을 가지므로 해당 월 데이터를
    DELETE한 후 Batch INSERT하는 것이 쿼리 횟수와 처리 복잡도 측면에서 더 효율적이라고 판단했습니다.
  • 불필요한 비교 연산을 제거하고 JPA의 saveAll() 배치 처리 기능을 활용하여 수집 처리 시간을 최소화

조회 성능 최적화: 쿼리 실행 시간 98% 단축 (45배 향상) [ 블로그 ]

  • 프로젝트의 핵심 기능인 '실거래가 검색'은, 데이터가 누적될수록 성능 저하가 발생할 수 있다고 판단 EXPLAIN, nGrinder, MySQL Profiling을 활용하여 문제를 진단하고, 인덱스 설계를 통해 성능을 개선
  • Full Table Scan의 발견
    • EXPLAIN 분석 결과, 약 17만 건의 데이터에 대해 type: ALL (Full Table Scan)이 발생하고 있었습니다.
    •  nGrinder를 통해 99명의 동시 접속 부하 테스트 시, 99명을 초과하면 에러가
      발생      하는 한계점을 확인했습니다.
  • 복합 인덱스 설계 및 적용
    • WHERE 절의 검색 패턴(동등 비교 우선, 범위 비교 후순)을 분석하여, 가장 효율적인 (sgg_cd, umd_nm, area) 복합 인덱스를 설계하고 적용
  • 개선 결과 : 데이터로 증명된 성능 향상

Spring Security 커스텀 로그인 실패 처리 [ 블로그 ]

  • Spring Security의 기본 로그인 실패 처리는 "자격 증명에 실패하였습니다" 같은 에러 메시지만을 제공
    사용자는 "ID가 틀린 것인지, 비밀번호가 틀린 것인지"에 대한 명확한 피드백을 주지 못하여 사용자 경험(UX)이 낮아지는 문제점이 있었습니다.
    • AuthenticationFailureHandler 인터페이스를 구현하는 LoginFailHandler 클래스를 직접 생성
    • onAuthenticationFailure 메서드를 오버라이드하여, 로그인 실패 시 발생하는 
      AuthenticationException의 구체적인 타입을 확인(instanceof)하는 분기 로직을 구현
    • UsernameNotFoundException 발생 시 "존재하지 않는 아이디입니다.", 
    • BadCredentialsException 발생 시 "비밀번호가 일치하지 않습니다." 와 같이, 각 예외 상황에 맞는 구체적인 에러 메시지를 설정
    • 마지막으로 SecurityConfig에 .failureHandler(loginFailHandler())를 등록하여
      Spring Security가 기본 핸들러 대신 직접 구현한 커스텀 핸들러를 사용하도록 설정했습니다.
  • 커스터마이징을 통해, 사용자는 로그인 실패 원인을 명확하게 인지하고 다음 행동을 결정할 수 있게 되어, 서비스의 사용 편의성을 크게 개선할 수 있었습니다.

쿠키 문자열 충돌 문제 [ 블로그 ]

  • 초기 구현 시 document.cookie.includes() 방식으로 hideNotice쿠키 존재 여부를 확인
  • 유사한 이름의 쿠키(do_not_hideNotice)가 존재할 경우 부분 문자열 일치로 오작동 가능성 발견
  • 해결 방안:
    • 쿠키 문자열을 ; 기준으로 분리
    • 각 항목이 정확히 "쿠키명="으로 시작하는지 검사하는 getCookie() 함수 구현
  • 다수 쿠키가 존재하는 실제 브라우저 환경에서도 안정적으로 동작하도록 개선

MyBatis 동적 쿼리 및 페이징 구현의 복잡성

  • 지역, 건물명, 보증금/월세 범위, 전용면적(평형) 등 사용자의 검색 조건은 매우 다양하며
    어떤 조건이 들어올지 예측할 수 없었습니다.
    Java 코드의 if-else 분기문으로 처리하는 것은 코드의 복잡도를 증가 시킬수 있다고 판단했습니다
    • 동적 쿼리 : MyBatis의 <if>, <choose>, <when> 등 동적 SQL 태그를 사용하여
      DTO의 필드 값이 null이 아닐 경우에만 WHERE 절에 AND 조건이 동적으로 추가되는 쿼리를 구현했습니다.
    • 파라미터 전달 : 여러 종류의 파라미터(검색조건 DTO, 페이징 변수)를 전달하기 위해
      @Param 어노테이션을 사용하여 각 파라미터의 이름과 타입을 명확히 지정하여 코드의 가독성을
      향상 시켰습니다.

개발/테스트 환경 구성 문제

  • 부하 테스트 도구인 nGrinder 실행 시, ClassNotFoundException이 발생하며 Worker Process가 시작되지 않는 문제 발생
  • 원인이 nGrinder 3.5.x 버전과 최신 Java(17+) 버전 간의 호환성 문제임을 파악했습니다.
    가장 안정적인 해결책은 테스트 실행 환경을 nGrinder가 공식적으로 지원하는 Java 11로 다운그레이드하는 것이라고 판단
    JAVA_HOME 환경 변수를 재설정하여 안정적인 테스트 환경을 구축하고 부하 테스트를 성공적으로 완료했습니다.

2️⃣ [ gidor_Helper ]

레거시 물류 시스템 업무 자동화 프로그램
GitHub : https://github.com/gi-dor/gidor_Helper

프로젝트 개요

2005년 이전부터 사용된 레거시 MSSQL 기반 물류 시스템에서

매일 반복적으로 수행되던 SQL 조회/수정 작업을 자동화하기 위해 개발한 업무 보조 프로그램입니다.

테이블명이 LS + 숫자 형식으로 구성되어 있고

관계 구조가 복잡해 매번 쿼리를 수동으로 작성해야 하는 비효율적인 환경이었고

이것을 회사 에서 사용하는 WinForms 을 이용해 UI 프로그램으로 자동화하여

업무 처리 속도와 정확성을 개선했습니다.

사용 기술

C# / .NET Framework / WinForms / MSSQL / Visual Studio

[ 주요 기능 ]

  • 🔎 사용자 및 영업소 정보 조건 조회 (ID, 사용자명, 고객사, 권한, 영업소 등 복합 검색)
  • 📦 운송장 스캔 데이터 관리
    • 최신 5,000건 조회
    • 상세 정보 확인 및 수정
    • 상태별 일괄 삭제 처리
  • 🗂 공통 코드 관리
    • 코드 구분 / 코드 내용 조건 검색
    • 동적 쿼리 기반 조회
  • 🏢 영업소 관리
    • 전체영업소 , 실제운영하는 영업소 , 폐쇄된 영업소 구분 조회
    • ComboBox 기반 선택 조회 기능 구현
