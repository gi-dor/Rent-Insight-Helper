프로젝트의 아키텍처 설명

왜 SpringBoot을 사용했는지 ??

- Spring 의 기능인 DI AOP 같은 기능을 그대로 사용하면서
  복잡한 설정 없이  사용할 수 있게 해주는 기술이다
  톰캣을 내장해서 별도의 웹서버를 설치하지 않아도된다 @**SpringBootApplication** 어노테이션이 선언된 클래스의 main() 메서드를 실행 하면 서버를 구동시킨다

DI ?

- DI 는 필요한 객체를 직접 `new`  키워드로 생성하지 않고 스프링 컨테이너가 대신 생성해서 주입해주는 방식입니다
  AController 에서 BService 가 필요한데 BService service = new Service 하지 않고 **생성자 주입 방식**으로 스프링에게 받아와 사용합니다

생성자 주입 방식으로는 어떻게 사용했나요 ?

- 컨트롤러에서 주입받을 서비스 클래스를 final 키워드를 사용하고 , 롬복의 @RequiredArgsConstructor 를 사용했습니다

@Autowired는 뭔가요 ?

- **필드에 직접 의존성을 주입하는 방식**입니다
- final 키워드를 사용이 불가능하며 코드의 길이가 간결해 집니다

AOP  ?

- 모든 메서드에 공통적으로 사용할 부가기능을 복사 붙여넣기 하지 않고 한곳에서 처리하는 기술입니다

SpringSecurity 동작 흐름 설명해주세요

- 사용자의 요청이 들어오면 DispatcherServlert 에 도달하기전에 Security가 설치한 필터를 거칩니다
- SecurityContextPersistenceFilter를 거쳐서 이전에 로그인을 해서 세션에 SecurityContext가 있다면 계속 인증된 상태를 유지시켜줍니다
- 사용자가 /login 경로로 아이디와 비밀번호를 입력해 로그인을 시도하면 UsernamePasswordAuthenticationFilter 가 요청을 가로채서 DB의 사용자 정보를 비교합니다
- 인증과정에서 AuthenticationManager 가 인증 실패를 결정하면
  제가 구현한 LoginFailHandler 에서 실패 케이스를 파악해야 사용자에게 에러메시지를 보여줍니다
- 인증에 성공한 사용자의 요청은 마지막 필터인 AuthorizationFilter를 거치는데 사용자가 권한이 있는지 체크하고 그 권한으로 해당 페이지에 접근할수 있는지 판단합니다
- 그 이후에 DispatcherServlert 을 거쳐 컨트롤러로 들어옵니다

UserDetailsService 는 뭐하는 인터페이스 인가 ?

- Spring Security에서 `데이터베이스에 저장된 실제 사용자 정보를 가져오는 역할`  을 담당합니다
- loadUserByUsername 라는 단하나의 추상 메서드만 정의되어있습니다
- 로그인 ID를 받아와 사용자 정보를 찾는 것입니다

왜 BCryptPasswordEncoder 를 사용했는지 ?

- 혹여나 발생할 DB 탈취 때문에 사용자의 비밀번호 암호화를 선택했고
- 단방향 이기 때문에 DB에 저장된 값을 가지고 원래의 비밀번호를 계산할 수 없어 선택했습니다

로그인 실패 처리는 어떻게 했는지 ?

- `AuthenticationFailureHandler` 를 구현한 LoginFailHandler 클래스를 만들었습니다
- `onAuthenticationFailure` 를 구현하면서 로그인 실패시 전달되는 Exception 객체의 타입을  `instanceof` 을 사용해 확인했고 각 예외별로 에러메시지를 설정했습니다
- SecurityConfig에 설정한 formLogin() 설정에서 failureHandler()를 사용해 제가 만든 loginFailHandler()를 등록해 로그인 실패시 기본동작이 아닌 제가 만든 로직을 따르게됩니다

세션 기반인가 JWT 기반인가

- Spring Security 기본설정을 그대로 사용해 로그인을 했을때 세션을 생성하고
  그 세션의 ID를 브라우저의 쿠키에 저장하게 했습니다

세션과 쿠키에 대해서 설명해주세요

- 쿠키 :  서버가 만들어서 웹 브라우저의 저장하는 작은 기록정보 파일
    - 사용자를 식별하기 위한 최소한의 정보
    - 서버가 설정한 말료일까지 유지된다
- 세션
    - 일정시간동안 같은 사용자로부터 들어오는 요구사항을 하나의 상태로 보고 그 상태를 유지시키는 기술
    - 일정시간은 사용자가 웹브라우저를 통해 접속한 시점부터 브라우저를 종료해서 연결을 끝내는 시점

- 세션 쿠키
    - 유효기간을 설정하지 않은 쿠키 , 해당 쿠키는 브라우저의 메모리에만 잠시 저장
    - 브라우저 종료시 쿠키는 사라진다

- 영속쿠키
    - 유효기간을 명시적으로 설정한 쿠키
    - 브라우저의 메모리가 아닌 사용자의 하드디스크에 파일 형태로 저장
    - 브라우저 종료를 해도 그대로 남아있는다

왜 세션기반을 했는지 ?

- 세션 같은 경우 Spring Security에서 기본으로 제공하는데, JWT를 배우고 적용하기엔 주어진 시간도 그렇고, 토큰의 생성 같은 모든 과정을 구현해야 해서 다른 쪽에 집중해야 하기에 Security 세션 방식을 선택했습니다

DELETE 후 INSERT 방식을 왜 선택했는지 ?

- 만약  종로구에서 1000건의 데이터가 존재하면 이 1000건의 데이터를 각각 DB에 존재하는지 확인하기위해 1000번에 SELECT  쿼리가 발생하게되고 작업시간이 늘어나 비효율이라 생각했습니다
- DELETE 후에 INSERT 를 하게되면 DB 를 사용하는 횟수가 줄어 전체적인 시간을 단축 시킬 것이라 생각했습니다

트랙잭션은 어디에 걸었는지 ?

- 데이터의 원자성과 일관성이 중요하다  생각해 데이터를 수집하는 메서드에 적용했습니다
- 10개 지역구는 성공하고 INSERT 했는데 11번째 지역구의 데이터를 처리하다가 에러가 발생했는데
  트랙잭션이 있다면 지금까지 진행한 DELETE , INSERT 모든 DB 작업을 자동으로 ROLL BACK 시키기 때문에 불안전한 데이터가 발생하지 않습니다

트랜잭션에 ACID 에 대해 말해주세요

- 원자성 : 모든 작업은 하나의 단위로 묶여 , 모두 성공하거나 모두 실패해야 합니다
- 일관성 : 트랜잭션이 성공적으로 완료된 이후 데이터베이스는 항상 일관 되어야 하는 상태입니다
- 고립성 : 여러개의 트랜잭션이 동시에 실행될때 , 다른 트랜잭션의 작업에 간섭받지 않아야 합니다
- 지속성: 성공적으로 완료된 트랜잭션 결과는 영구적으로 데이터베이스에 기록되고 보존되어야 합니다

JPA vs MyBatis 차이와 왜 그렇게 두개를 사용했는지 설명해주세요

- JPA 같은경우 SQL 쿼리를 직접 장성하지 않는 대신 자바객체 Entity 에 집중하게됩니다
    - findById() , save() 같은 기본적인 CRUD 쿼리는 직접 작성없이 JPA가 알아서 생성해줍니다
- MyBatis는 SQL 쿼리를 직접 장성하고 실행의 결과매핑을 도와줍니다
    - 익숙한 SQL 문법을 사용하여 복잡한 조건의 쿼리를 제어할수 있습니다
    - 특히 동적으로 IF  , CHOOSE 같은 SQL 태크를 사용할수 있습니다
    - 사용자가 어떤 검색조건을 입력하든 그에 맞는 WHERE 절을 조합하여 작성할 수 있습니다

인덱스는 왜 필요한지 ?

- 인덱스가 없을때는 Full Table Scan으로 테이블의 첫번째 부터 마지막까지 모든 행을 다 읽어서 비교해야합니다
- 인덱스가 있다면 Index Scan으로 인덱스를 설정한 컬럼으로 데이터베이스는 해당 컬럼의 값으로 정렬하여 전체 테이블을 조회하지 않습니다

팝업 모달창에 24시간 보지 않기 기능은 어떻게 구현했는지 ?

- 닫기 버튼을 누르는 순간 체크박스가 선택되었는지 확인하고 선택되었다면 setCookie() 함수 실행
- 이때 매개변수로 hideNotice , true , 1이 넘어갑니다 쿠키의 이름 , 값 , 유효기간을 의미합니다
- setCookie()함수에서 유효기간을 계산하고 영속쿠키를 사용자의 브라우저에 생성해줍니다

- 사용자가 페이지에 접속할때 getCookie() 함수를 사용해 해당 쿠키가 존재하는지 확인합니다
- 다른 쿠키들과 문자열 방지를 위해 `;`  기준으로 분리해서 검사합니다

왜 includes() 방식은 위험한지 ?

- 기존 includes(’hideNotice’) 를 사용하게되면 쿠키 이름이 dp-not-hideNotice 가 있다면
  포함되므로 true 를 반환 하기때문에 다시보지 않기를 선택하지 않아도 팝업을 보여주지 않기에
  `;` 을 기준으로 분리해서 각 조각별로 내가 찾는 쿠키이름을 검사해서 다른 쿠키와의 충돌을 차단합니다

외부 API 키는 어떻게 관리 했는지 ?

- application.yml 파일에 직접 키를 입력하지 않고
- application-local.yml 파일에 실제로 키를 입력하고 .gitignore에 추가 했습니다
    - 깃허브에 커밋되지 않게하고 Java 코드에서는 @Value 어노테이션을 사용해 설정파일의 값을 주입받습니다

서버에서 호출하나 ? , 프론트에서 호출하나 ? 왜 그렇게 설계했는지 ?

- 서버에서 API를 호출합니다 @Scheduled 붙은 CallApiService 에서 시작하게되는데
- 서비스 키는 외부에 노출되어서는 안되는 정보입니다
- 만약 프론트엔드에서 API를 호출하게되면 JS 안에 그대로 노출되기에 ‘ 소스보기’  기능으로 서비스 키 가 노출될수 있습니다
